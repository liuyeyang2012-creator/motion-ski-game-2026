# 姿态识别组件加载恢复设计

## 1. 问题

手机摄像头已经正常开启，但页面永久停留在“识别组件加载中”。当前 `createDirectPoseClient()` 依次等待 MediaPipe WASM 文件选择和 Pose Landmarker 初始化，两段异步过程都没有截止时间。只要手机浏览器在 WASM 编译、模型初始化或设备兼容性判断中不返回，校准页就不会成功，也不会进入失败状态。

线上首页、模型文件、SIMD 与非 SIMD WASM 文件均可正常访问，因此本次修复聚焦客户端初始化的有界等待和兼容降级，不更换姿态模型，也不修改已确认的动作校准规则。

## 2. 方案选择

采用“普通模式超时 + 非 SIMD CPU 兼容模式”的方案：

- 仅增加超时可以避免永久卡死，但同一手机重试后仍可能再次失败。
- 普通模式超时后切换非 SIMD CPU 路径，可以避开部分安卓浏览器的 SIMD 或加速兼容问题。
- 将全部识别迁入 Web Worker 隔离更彻底，但涉及视频帧传输、生命周期和性能重构，不适合作为本次紧急修复。

## 3. 加载流程

1. 摄像头开启后进入普通识别组件加载。
2. 普通模式最多等待 15 秒。
3. 15 秒内成功时进入人体识别，流程与当前版本一致。
4. 普通模式抛错或超时时，页面进入明确的失败状态，不再无限显示加载中。
5. 失败页主按钮显示“兼容模式重试”。
6. 兼容模式使用 `vision_wasm_nosimd_internal.js`、`vision_wasm_nosimd_internal.wasm`，并明确指定 CPU delegate。
7. 兼容模式同样最多等待 15 秒。
8. 兼容模式成功后进入人体识别；失败后仍停留在可恢复页面，按钮显示“再次尝试兼容模式”。

## 4. 资源与生命周期

- 切换普通模式和兼容模式时保持当前摄像头流，不再次申请摄像头权限。
- 每次新尝试开始前关闭已经创建的旧 Pose Landmarker。
- 超时的初始化任务若稍后才返回，立即关闭其 Pose Landmarker，不得覆盖当前尝试。
- 页面切后台、横屏中断或开始新的校准时，旧初始化结果不得恢复旧界面。
- 重试按钮一次只启动一个初始化任务，连续点击不会创建并行模型实例。

## 5. 界面反馈

- 普通加载：显示“识别组件加载中”，并补充“首次加载可能需要一些时间”。
- 普通失败：显示“普通模式未能启动”，主按钮为“兼容模式重试”。
- 兼容加载：显示“兼容模式加载中”。
- 兼容失败：显示“兼容模式未能启动”，主按钮为“再次尝试兼容模式”。
- 加载失败不关闭摄像头，玩家仍能看到取景画面。

## 6. 技术边界

- `direct-pose-client.ts` 负责根据 `standard | compatibility` 选择 WASM fileset，并在兼容模式下指定 CPU。
- `app-controller.ts` 负责 15 秒超时、迟到结果清理、当前尝试校验和重试模式切换。
- `calibration-session.ts` 记录当前加载模式和失败状态，供界面生成准确文案。
- 不增加远程服务、日志上传或摄像头数据上传。

## 7. 验收标准

- 任意模型初始化最多 15 秒，页面不会永久停留在加载中。
- 普通模式失败后可在不重新授权摄像头的情况下进入兼容模式。
- 兼容模式使用非 SIMD WASM 文件并明确使用 CPU delegate。
- 超时后迟到的模型实例被关闭，不能覆盖新尝试。
- 多次点击重试不会创建并行初始化。
- 模型成功后仍能进入现有的人体识别与动作校准流程。
- 现有 104 项测试、类型检查、Pages 构建和手机端 E2E 路径继续通过。

## 8. 测试范围

- 单元测试覆盖普通与兼容模式的资源路径及 CPU delegate。
- 控制器测试使用永不完成的 Promise 复现截图中的永久等待，并用假计时器确认 15 秒后进入失败状态。
- 控制器测试覆盖兼容重试不重新启动摄像头。
- 控制器测试覆盖超时任务迟到返回时被关闭。
- 界面测试覆盖普通加载、普通失败、兼容加载和兼容失败文案。
- 完整测试与构建通过后重新发布 GitHub Pages。
